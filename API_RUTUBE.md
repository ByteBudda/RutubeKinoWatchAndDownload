
# 🛸 Rutube API: Полная техническая спецификация (v2.1)
Эта документация описывает «дикий» API Rutube, полученный методом реверс-инжиниринга. Она предназначена для создания альтернативных клиентов и систем автоматизации.
## 1. Фундаментальные основы
### 1.1 Базовые параметры
 * **Base URL:** https://rutube.ru/api/
 * **Протокол:** HTTPS (HTTP/2 приветствуется)
 * **Формат данных:** Строгий JSON.
### 1.2 Заголовки (The "Must-Have" Headers)
Rutube крайне чувствителен к User-Agent. Использование браузерного UA часто ведет к проверкам на ботов (Cloudflare/Captcha).
 * User-Agent: Mozilla/5.0 (SmartTV; Tizen) AppleWebKit/537.36 — **Золотой стандарт.** Имитирует Smart TV, что снижает вероятность блокировок и часто упрощает структуру JSON.
 * Accept: application/json — Обязателен для корректной интерпретации ответов.
 * Referer: https://rutube.ru/ — Критичен при запросе прямых ссылок на видео (get_play_options).
## 2. Типология эндпоинтов (ДНК системы)
Rutube API построен не на строгих схемах, а на «матрешках». Мы выделяем 4 ключевых типа:
### 2.1 Контейнер-Витрина (Category/Feed)
**Пример:** /api/feeds/movies/
Это высокоуровневый эндпоинт, который не содержит видео, а описывает структуру страницы.
 * **Ключевой маркер:** Наличие массива tabs.
 * **Логика:** Каждая вкладка (tab) содержит массив resources. Каждый ресурс — это ссылка на следующий эндпоинт.
 * **Комментарий:** Это «карта дорог». Здесь мы строим главное меню приложения.
### 2.2 Группа-Прослойка (Cardgroup)
**Пример:** /api/feeds/cardgroup/1085
Самый коварный тип. Может вести себя и как папка, и как список видео.
 * **Логика:** Если results пуст, ищем ссылки в other_links или в теле виджетов. Если results полон — работаем как с плейлистом.
 * **Комментарий:** Используется для тематических подборок внутри категорий (например, «Ужасы»).
### 2.3 Конечный Плейлист (Playlist/Tag)
**Пример:** /api/tags/video/7487/
Прямой список контента.
 * **Маркеры:** results (массив объектов), has_next (флаг пагинации).
 * **Данные:** Содержит миниатюры, заголовки, длительность и author_id.
### 2.4 Метаданные Видео (Video Object)
**Пример:** /api/video/{hash_id}/
Детальная информация об одном ролике.
## 3. Анатомия объекта Video (Results Item)
Каждый элемент в массиве results — это готовая карточка видео.
| Поле | Тип | Комментарий |
|---|---|---|
| id | string | **Хеш-ID (32 символа).** Не путать с числовым ID. Используется для всех плееров. |
| title | string | Заголовок видео. Бывает обрезан на стороне сервера. |
| thumbnail_url | string | Прямая ссылка на постер. Обычно .jpg или .webp. |
| duration | int | Длительность в **секундах**. |
| author | object | Содержит name и id канала. Важно для фильтрации (например, только Wink). |
| is_paid | bool | Если true — видео требует подписки. yt-dlp может не сработать. |
| embed_url | string | Ссылка для <iframe>. Формат: https://rutube.ru/play/embed/{id}. |
## 4. Пагинация и навигация
Rutube использует классическую курсорную пагинацию.
 * **Параметры запроса:**
   * page=N (по умолчанию 1).
   * limit=N или per_page=N (рекомендуется 50-100 для скорости).
 * **Поля ответа:**
   * has_next: true/false.
   * next: Полный URL следующей страницы. **Внимание:** URL в next часто содержит лишние параметры, которые ломают ваш локальный кэш. Требует нормализации (как в твоем краулере).
## 5. Специфика "Кинотеатра" (Metainfo & TV)
Работа с сериалами отличается от обычных видео.
 1. **Список шоу:** /api/metainfo/tv/
 2. **Сезоны:** /api/metainfo/tv/{tv_id}/seasons/
 3. **Серии:** /api/metainfo/tv/{tv_id}/video/?season={N}
   * *Нюанс:* Если не передать season, API может отдать последние загруженные серии из всех сезонов вперемешку.
## 6. Алгоритм "Умного обхода" (Drill-Down)
Чтобы ваш API-клиент работал плавно, он должен следовать этой логике при получении любого URL:
 1. **Запрос:** Получаем JSON.
 2. **Классификация:**
   * Видим tabs? -> Выводим меню вкладок.
   * Видим results с видео? -> Выводим сетку видео.
   * Видим results пустой, но есть other_links? -> Переходим по первой ссылке (Auto-drilldown).
 3. **Кэширование:** Сохраняем результат в NodeCache по нормализованному ключу (URL без page).
## 7. Обработка ошибок и лимиты (Rate Limiting)
 * **HTTP 429:** Вы превысили лимит. Rutube «банит» на время от 30 секунд до 5 минут.
   * *Решение:* Экспоненциальная задержка (пауза 1с -> 2с -> 4с).
 * **HTTP 401/403:** Видео защищено или платно.
 * **HTTP 404:** Видео удалено или эндпоинт устарел (у Rutube часто «протухают» ссылки в cardgroup).
## 8. Практические советы
 1. **Нормализация:** Всегда удаляйте параметры _ (timestamp) и дубликаты limit из приходящих от API ссылок перед сохранением в базу.
 2. **Фильтрация мусора:** В ответах часто приходят блоки advertising и analytics. Смело вырезайте их на уровне прокси-сервера, чтобы не перегружать память клиента.
 3. **Поиск:** Эндпоинт /api/search/video/?query={q} самый стабильный. Если краулер зашел в тупик — поиск всегда вернет плоский список results.
**Вердикт:** Rutube API — это непредсказуемая, но богатая ресурсами среда. Имея на руках эту документацию и твой дамп, можно построить клиент, который будет стабильнее и быстрее официального сайта.
